Setup guide

From zero to production-ready in 5 minutes

Everything installs through the Claude Code CLI. No manual config files to write.

1

Clone the template

This gives you CLAUDE.md, TASKS.md, and .gitignore — the project-specific layer that ECC doesn't provide.

git clone https://github.com/mboss37/claude-launchpad.git my-project
cd my-project
rm -rf .git && git init
2

Install the ECC plugin

This installs all 16 agents, 65+ skills, 40+ commands, and 20 hooks in one shot. Run these inside Claude Code:

# Start Claude Code
claude

# Add the marketplace and install
> /plugin marketplace add affaan-m/everything-claude-code
> /plugin install everything-claude-code@everything-claude-code

# Restart Claude Code to load the plugin
> /exit
claude

You'll be prompted to choose scope: user (all projects), project (collaborators), or local (you only, this repo).

Note: Plugins are runtime extensions — they don't copy files into your project. After restart, all skills, agents, hooks, and commands are available automatically. You won't see them in your .claude/ folder, and that's correct. They live in ~/.claude/plugins/.

3

Install language rules

Rules are conventions that load automatically based on file type. Pick the language(s) for your project. This installs common rules (9 files) plus language-specific rules (5 files each).

# Clone the ECC repo (just for the install script)
git clone https://github.com/affaan-m/everything-claude-code.git /tmp/ecc

# Install rules for your language(s)
cd /tmp/ecc && ./install.sh <language>

# Options:
#   ./install.sh typescript        # Next.js, React, Node.js
#   ./install.sh python            # FastAPI, Django, Flask
#   ./install.sh golang            # Go projects
#   ./install.sh swift             # iOS/macOS
#   ./install.sh typescript python # Multiple languages

Rules install to ~/.claude/rules/ (global). They apply to all your projects.

4

Configure CLAUDE.md for your stack

Open CLAUDE.md and fill in the TODO sections — or just tell Claude your stack and let it do it:

# Start Claude Code
claude

# Tell it your stack
> "I'm building a SaaS app with Next.js, shadcn/ui, and Supabase.
   Fill in CLAUDE.md for this stack."

Claude rewrites the stack, architecture, commands, project structure, and conventions sections automatically.

5

Start building

You now have the full ECC toolkit. Use slash commands to drive your workflow:

> /plan "Build user authentication with email/password and OAuth"
# Creates a phased plan, waits for your confirmation

> /tdd
# Writes tests first, then implements

> /verify pre-commit
# Runs build + lint + tests + security checks

> /code-review
# Reviews all uncommitted changes
How it works

Two layers, one setup

Your project's memory, conventions, and task tracking. Committed to git, shared with your team.

  • TASKS.md — sprint tracker with session continuity
  • .env.example — required environment variables
  • .gitignore — sensible defaults
ECC Plugin

Agent toolkit layer

Production-grade agents, skills, hooks, and rules. Installed as a plugin, managed separately.

  • 16 agents — planner, architect, TDD, code review, security
  • 65+ skills — patterns, workflows, and framework guides
  • 40+ commands — slash commands for every workflow
  • 20 hooks — auto-format, type check, security gates
  • Per-language rules — TypeScript, Python, Go, Swift
Typical workflow

How a feature gets built

ECC enforces a structured development pipeline. Each phase uses specialized agents and commands.

/plan Plan
→
/tdd Write tests
→
Implement
→
/verify Verify
→
/code-review Review
→
Commit

1. Plan — /plan restates requirements, identifies risks, and creates a phased implementation plan. Waits for your confirmation before any code is written.
2. Test first — /tdd scaffolds interfaces and writes failing tests (RED), then implements minimal code to pass (GREEN), then refactors (IMPROVE). Targets 80%+ coverage.
3. Implement — Write the feature. ECC hooks auto-format your code, run type checks, and warn about console.log statements as you work.
4. Verify — /verify runs build, type check, lint, tests, and console.log audit in one command. Supports quick, full, pre-commit, and pre-pr modes.
5. Review — /code-review checks for security vulnerabilities, code quality, performance issues, and missing tests.
6. Commit — Conventional commits enforced by rules. Hooks remind you to review before pushing.

20 Hooks

Automated quality gates

Hooks fire automatically before or after Claude uses tools. They format code, check types, block dangerous commands, and capture learning — all without you doing anything.

Before tool execution (PreToolUse)

These run before Claude executes a command. Some warn, some block entirely.

PreToolUse • BLOCKS

Dev server blocker

Blocks npm run dev and similar outside tmux, ensuring log access for dev servers.

PreToolUse • WARNS

Tmux reminder

Suggests tmux for long-running commands like npm test, cargo build, docker.

PreToolUse • WARNS

Git push reminder

Reminds you to review changes before git push.

PreToolUse • WARNS

Doc file warning

Warns about creating non-standard documentation files. Allows README, CLAUDE, CHANGELOG, etc.

PreToolUse • SUGGESTS

Strategic compact

Suggests manual /compact at logical intervals (~50 tool calls) to preserve context quality.

After tool execution (PostToolUse)

These run after edits, builds, and other tool calls. Mostly auto-formatting and quality checks.

PostToolUse

Prettier / Biome format

Auto-formats JS/TS files after every edit. Auto-detects Biome or Prettier.

PostToolUse

TypeScript type check

Runs tsc --noEmit after editing .ts/.tsx files to catch type errors immediately.

PostToolUse

console.log warning

Warns about console.log statements left in edited files.

PostToolUse

PR logger

Logs PR URL and review command after gh pr create.

PostToolUse • ASYNC

Build analysis

Background analysis after build commands. Non-blocking, 30s timeout.

PostToolUse • ASYNC

Quality gate

Runs fast quality checks after file edits. Non-blocking, 30s timeout.

Session lifecycle (Stop / Start / Compact)

Stop

Console.log audit

Checks all modified files for console.log at session end.

Stop • ASYNC

Pattern extraction

Evaluates the session for extractable patterns (continuous learning).

Stop • ASYNC

Session summary

Persists session state for continuity across restarts.

PreCompact

State saver

Saves important state before context compaction so nothing is lost.

Hook profiles: Control which hooks run via ECC_HOOK_PROFILE=minimal|standard|strict. Disable specific hooks with ECC_DISABLED_HOOKS=hook-id-1,hook-id-2.
Rules

Conventions that enforce themselves

Rules are loaded into context automatically based on file type. Common rules apply to all projects; language rules add framework-specific conventions on top.

Common rules (all projects)

9 rule files covering universal development standards. Installed with every language.

coding-style

  • Immutability — new objects, never mutate
  • Small files (200-400 lines, 800 max)
  • Functions under 50 lines
  • No deep nesting (>4 levels)

security

  • No hardcoded secrets
  • SQL injection prevention
  • XSS prevention
  • CSRF, rate limiting, auth checks

testing

  • 80% minimum coverage
  • Unit + integration + E2E required
  • TDD workflow enforced
  • Fix implementation, not tests

git-workflow

  • Conventional commits format
  • Analyze full commit history for PRs
  • Comprehensive PR summaries

development-workflow

  • Research & reuse first
  • Plan before coding
  • TDD approach mandatory
  • Code review after implementation

performance

  • Model selection (Haiku/Sonnet/Opus)
  • Context window management
  • Extended thinking usage

agents

  • When to use each agent
  • Parallel task execution
  • Multi-perspective analysis

patterns

  • Skeleton project workflow
  • Repository pattern
  • API response envelope format

hooks

  • Hook types and lifecycle
  • Auto-accept permissions
  • TodoWrite best practices
